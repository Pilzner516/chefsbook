=== Ralph Session: Debug Done Chef Button - Menu Cook Flow ===
Started: 2026-05-05
Task: Identify root cause of Done Chef button and TTS failure in menu cook flow
Status: ✅ ROOT CAUSE IDENTIFIED

## ROOT CAUSE ANALYSIS

### Database Evidence
Examined stuck cooking sessions:
- Session ID: 07630972-ee8f-4d4b-9e63-dd996501df81
- Status: 'cooking'
- current_step_index: 0 (stuck, never advances)
- version: 2 (successfully transitioned briefing → cooking)
- step_actuals: [] (empty - no steps ever recorded as complete)
- User ID: b589743b-99bd-4f55-983a-c31f5167c425

### Code Comparison: Mobile vs Web

**Mobile implementation (apps/mobile/app/cook-menu/[id]/active.tsx) - CORRECT:**
```typescript
const { success, session: updated } = await updateCookingSession(
  session.id,
  { current_step_index: nextIndex, step_actuals: newActuals, ... },
  session.version  // ← Simple: always uses current session.version
);

if (!success || !updated) {
  // Version conflict — refetch fresh session from DB
  const fresh = await getCookingSession(session.id);
  if (fresh) {
    setSession(fresh);
    showToast('Another device updated');
    resetTimerForIndex(fresh, fresh.current_step_index);
  }
  return;  // ← STOPS here, user can retry
}
```

**Web implementation (apps/web/app/dashboard/menus/[id]/cook/active/page.tsx) - BUGGY:**
```typescript
// Line 220-224: Check return value? Nope!
if (overrunMinutes > 2) {
  updatedPlan = recomputeFromOverrun(session.plan, currentStep.step.id, actualEnd);
  await persistRecomputedPlan(session.id, updatedPlan, session.version);
  // ← Doesn't check if persistRecomputedPlan succeeded!
  // ← Comment says "fetch fresh" but doesn't actually fetch
}

// Line 228-238: Wrong version calculation + silent failure
const result = await updateCookingSession(
  session.id,
  { current_step_index: nextIndex, step_actuals: updatedActuals, ... },
  overrunMinutes > 2 ? session.version + 1 : session.version
  // ↑ BUG: When overrun > 2, version already incremented by persistRecomputedPlan,
  //       but using stale session.version from closure
);

if (result.session) {
  setSession(result.session);
  // ... continues
}
// ← BUG: No else clause! Silent failure, no refetch, no retry, no user feedback
```

### The Three Bugs

**Bug 1: Ignored persistRecomputedPlan failure**
- persistRecomputedPlan returns boolean indicating success/failure
- Web code doesn't check the return value (line 222)
- If version conflict occurs during plan persist, code continues with wrong assumptions

**Bug 2: Version conflict race condition**
- Realtime subscription updates session state asynchronously
- persistRecomputedPlan increments version in DB
- Second updateCookingSession uses stale session.version from closure
- Version mismatch → optimistic lock failure → silent exit

**Bug 3: No error recovery**
- When updateCookingSession returns `{ success: false, session: null }`, code checks `if (result.session)`
- Condition is false → skips the block → function exits silently
- User sees no error, no feedback, button click appears to do nothing
- No refetch, no retry logic, no toast/alert

### Failure Scenario (Exact sequence)

1. User loads page, session at version 2, step 0
2. Realtime subscription fires in background (another tab/device), session updates to version 3
3. React state updated, but handleStepComplete closure captures old session (version 2)
4. User clicks "Done Chef"
5. calculateOverrun() determines overrunMinutes > 2
6. persistRecomputedPlan called with expectedVersion=2, DB is version 3
7. persistRecomputedPlan fails (PGRST116 - no rows matched), returns false
8. **Web code ignores failure, continues**
9. updateCookingSession called with expectedVersion=3 (session.version + 1 = 2 + 1)
10. But DB is still version 3 (persistRecomputedPlan failed)
11. Update fails: expected version 3, needs to match and increment to 4
12. Returns { success: false, session: null }
13. `if (result.session)` check fails
14. Function exits silently
15. User sees nothing, step stays at 0, step_actuals stays empty

### Why Mobile Works
- Always uses simple `session.version` (no conditional)
- Explicitly checks `if (!success || !updated)`
- Refetches fresh session on conflict: `getCookingSession(session.id)`
- Shows user feedback: `showToast('Another device updated')`
- Resets timer and UI state with fresh data
- User can immediately retry with correct version

### The Fix (Port mobile pattern to web)
1. Remove complex version conditional
2. Always use `session.version` directly
3. Check persistRecomputedPlan return value, refetch if failed
4. Add proper error handling: check `!success || !result.session`
5. Refetch fresh session on conflict
6. Show user feedback (alert/toast)
7. Reset UI state with fresh data

### Files Requiring Fix
- apps/web/app/dashboard/menus/[id]/cook/active/page.tsx lines 189-259
